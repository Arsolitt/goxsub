# Overview

sing-box is the universal proxy platform, written in Go.

## Key Features

- **Multi-protocol support**: Shadowsocks, VMess, VLESS, Trojan, Hysteria, Hysteria2, Naive, TUIC, ShadowTLS, AnyTLS, WireGuard, Tor, SSH, SOCKS, HTTP
- **TUN-based transparent proxy**: Virtual network interface that captures all system traffic (L3 to L4 conversion)
- **DNS routing**: Advanced DNS resolution with rule-based server selection, FakeIP support, and reverse mapping
- **Rule-based traffic splitting**: Match traffic by domain, IP, process, port, protocol, geosite/geoip, and more
- **Clash API compatible**: Experimental Clash API for dashboard integration
- **Configuration management**: JSON config with multi-file merge support, format checking, and SIGHUP reload

## Supported Platforms

| Platform | Support |
|----------|---------|
| Linux | Full support |
| macOS | Full support |
| Windows | Full support |
| Android | Full support (via graphical client) |
| iOS | Full support (via graphical client) |
| FreeBSD | Full support |

## Architecture

Data flows through sing-box as follows:

1. **Inbounds** accept connections (TUN, SOCKS, HTTP, protocol listeners)
2. **Router** matches traffic against route rules
3. **DNS Router** resolves domains using configured DNS servers and rules
4. **Outbounds** forward traffic to destinations (proxy servers, direct, block)

```
[Inbounds] -> [Router] -> [Outbounds]
                   |
              [DNS Router]
```

## Configuration Format

sing-box uses JSON for configuration. The top-level structure:

```json
{
  "log": {},
  "dns": {},
  "ntp": {},
  "certificate": {},
  "endpoints": [],
  "inbounds": [],
  "outbounds": [],
  "route": {},
  "services": [],
  "experimental": {}
}
```

## Important Notes

- **Tags must be unique** across all inbounds, outbounds, and endpoints (shared namespace)
- **Unknown JSON fields are rejected** by the parser (no silent ignoring)
- **Configuration is JSON only** - no YAML, TOML, or INI
- **Config merging**: Multiple config files can be merged; arrays are concatenated, objects are deep-merged

# Installation

## Repository Installation

### Debian / APT

```shell
sudo mkdir -p /etc/apt/keyrings &&
   sudo curl -fsSL https://sing-box.app/gpg.key -o /etc/apt/keyrings/sagernet.asc &&
   sudo chmod a+r /etc/apt/keyrings/sagernet.asc &&
   echo '
Types: deb
URIs: https://deb.sagernet.org/
Suites: *
Components: *
Enabled: yes
Signed-By: /etc/apt/keyrings/sagernet.asc
' | sudo tee /etc/apt/sources.list.d/sagernet.sources &&
   sudo apt-get update &&
   sudo apt-get install sing-box
```

### RedHat / DNF

```shell
sudo dnf config-manager addrepo --from-repofile=https://sing-box.app/sing-box.repo &&
sudo dnf install sing-box
```

# Quick Start

## Minimal Client Configuration

A basic sing-box client that proxies all traffic through a SOCKS5 inbound:

```json
{
  "inbounds": [
    {
      "type": "mixed",
      "tag": "mixed-in",
      "listen": "127.0.0.1",
      "listen_port": 2080
    }
  ],
  "outbounds": [
    {
      "type": "shadowsocks",
      "tag": "proxy",
      "server": "server.example.com",
      "server_port": 8388,
      "method": "2022-blake3-aes-128-gcm",
      "password": "<base64-encoded-key>"
    },
    {
      "type": "direct",
      "tag": "direct"
    }
  ],
  "route": {
    "final": "proxy"
  }
}
```

## TUN Transparent Proxy Client

A more complete client using TUN for transparent proxy:

```json
{
  "dns": {
    "servers": [
      {
        "tag": "remote",
        "type": "tls",
        "server": "8.8.8.8"
      },
      {
        "tag": "local",
        "type": "udp",
        "server": "223.5.5.5"
      }
    ],
    "strategy": "prefer_ipv4"
  },
  "inbounds": [
    {
      "type": "tun",
      "tag": "tun-in",
      "address": ["172.19.0.1/30"],
      "auto_route": true,
      "strict_route": true
    }
  ],
  "outbounds": [
    {
      "type": "shadowsocks",
      "tag": "proxy",
      "server": "server.example.com",
      "server_port": 8388,
      "method": "2022-blake3-aes-128-gcm",
      "password": "<base64-encoded-key>"
    },
    {
      "type": "direct",
      "tag": "direct"
    }
  ],
  "route": {
    "rules": [
      {
        "action": "sniff"
      },
      {
        "protocol": "dns",
        "action": "hijack-dns"
      },
      {
        "ip_is_private": true,
        "outbound": "direct"
      }
    ],
    "default_domain_resolver": "local",
    "auto_detect_interface": true,
    "final": "proxy"
  }
}
```

### How it works

1. **TUN inbound** creates a virtual network interface that captures all system traffic
2. **Route rule `sniff`** enables protocol detection to identify the real destination
3. **Route rule `hijack-dns`** intercepts DNS queries for proper resolution
4. **Route rule `ip_is_private`** sends private IP traffic directly
5. **`final: "proxy"`** sends all remaining traffic through the proxy outbound
6. **`default_domain_resolver: "local"`** resolves outbound server domains using the local DNS

## Check Configuration

```shell
sing-box check -c config.json
```

Validates the configuration without starting the service.

## Run

```shell
sing-box run -c config.json
```

Starts the proxy service. Send `SIGHUP` to reload configuration.

## Format Configuration

```shell
sing-box format -w -c config.json
```

Formats and rewrites the config file with proper JSON indentation.

## Merge Configurations

```shell
sing-box merge output.json -c base.json -D config.d/
```

Merges multiple config files into one. The `-D` flag reads all `.json` files from a directory, sorted by filename.

## Working with Multiple Config Files

You can split configuration across multiple files in a directory:

```
config.d/
  01-dns.json
  02-inbounds.json
  03-outbounds.json
  04-route.json
```

Then run with:

```shell
sing-box run -D config.d/
```

Files are sorted alphabetically and deep-merged. Arrays are concatenated.

# Configuration Overview

sing-box uses JSON format for configuration. The configuration file contains all settings for logging, DNS, inbounds, outbounds, routing, and services.

## Top-Level Fields

> `log`: object - Log options
> `dns`: object - DNS options
> `ntp`: object - NTP options
> `certificate`: object - Certificate options
> `endpoints`: array - Endpoint list (share namespace with outbounds)
> `inbounds`: array - Inbound list
> `outbounds`: array - Outbound list
> `route`: object - Route options
> `services`: array - Service list
> `experimental`: object - Experimental options
> `$schema`: string - JSON schema URL (optional)

## Important Notes

- **JSON Format**: Configuration uses standard JSON format
- **Arrays**: `inbounds`, `outbounds`, `endpoints`, and `services` are arrays
- **Objects**: Settings like `log`, `dns`, `route` are objects
- **Unique Tags**: Tags must be unique across inbounds, outbounds, and endpoints (they share the same namespace)
- **Strict Parsing**: Unknown fields are rejected by the parser - this helps catch configuration errors
- **Optional Fields**: Most fields are optional and have sensible defaults

## Minimal Configuration Example

```json
{
  "$schema": "https://sing-box.sagernet.org/schema/config.json",
  "log": {
    "level": "info"
  },
  "inbounds": [
    {
      "type": "tun",
      "tag": "tun-in",
      "inet4_address": "172.19.0.1/30"
    }
  ],
  "outbounds": [
    {
      "type": "direct",
      "tag": "direct"
    }
  ]
}
```

## Configuration Structure

Each inbound and outbound requires:
- `type`: The protocol type (e.g., "tun", "socks", "http", "shadowsocks")
- `tag`: Unique identifier used for routing (must be unique across all inbounds/outbounds/endpoints)

Additional fields depend on the specific protocol type.

# Shared: Listen Fields

These fields are available on all inbound types for configuring how the inbound listens for connections.

## Listen Options

> `listen`: string - Listen address (e.g., "127.0.0.1", "::", "0.0.0.0")
> `listen_port`: number - Listen port (0-65535)
> `bind_interface`: string - Bind to specific network interface (e.g., "eth0")
> `routing_mark`: number - Routing mark (fwmark for policy routing, supports hex like `0xff`)
> `reuse_addr`: bool - Enable SO_REUSEADDR socket option
> `netns`: string - Network namespace path (Linux only)
> `disable_tcp_keep_alive`: bool - Disable TCP keepalive probes
> `tcp_keep_alive`: duration - TCP keepalive interval (e.g., "5m", "30s")
> `tcp_keep_alive_interval`: duration - TCP keepalive interval between probes
> `tcp_fast_open`: bool - Enable TCP Fast Open (TFO)
> `tcp_multi_path`: bool - Enable Multipath TCP (MPTCP)
> `udp_fragment`: bool - Enable UDP fragmentation (default depends on platform)
> `udp_timeout`: duration - UDP NAT timeout (accepts seconds as number or duration string like "5m")
> `detour`: string - Outbound tag to route listener traffic through

## Duration Format

Duration fields accept Go duration strings:
- `"30s"` - 30 seconds
- `"5m"` - 5 minutes
- `"1h"` - 1 hour
- `300` - 300 seconds (plain number for backward compatibility)

## Example

```json
{
  "type": "socks",
  "tag": "socks-in",
  "listen": "127.0.0.1",
  "listen_port": 1080,
  "tcp_fast_open": true,
  "udp_timeout": "5m"
}
```

## Routing Mark

The `routing_mark` field sets the fwmark on packets, useful for policy-based routing:

```json
{
  "type": "tun",
  "tag": "tun-in",
  "routing_mark": 0xff,
  "inet4_address": "172.19.0.1/30"
}
```

## Network Namespace

On Linux, you can specify a network namespace:

```json
{
  "type": "http",
  "tag": "http-in",
  "netns": "/var/run/netns/vpn"
}
```

# Shared: Dial Fields

These fields are available on all outbound types for configuring how the outbound establishes connections.

## Dialer Options

> `detour`: string - Use another outbound as the dialer (chain through another proxy)
> `bind_interface`: string - Bind to specific network interface (e.g., "eth0")
> `inet4_bind_address`: string - Bind to specific IPv4 address
> `inet6_bind_address`: string - Bind to specific IPv6 address
> `bind_address_no_port`: bool - Bind address without specifying port
> `protect_path`: string - Protect socket path (Android VPN protection)
> `routing_mark`: number - Routing mark (fwmark for policy routing)
> `reuse_addr`: bool - Enable SO_REUSEADDR socket option
> `netns`: string - Network namespace path (Linux only)
> `connect_timeout`: duration - Connection timeout (e.g., "5s")
> `tcp_fast_open`: bool - Enable TCP Fast Open (TFO)
> `tcp_multi_path`: bool - Enable Multipath TCP (MPTCP)
> `disable_tcp_keep_alive`: bool - Disable TCP keepalive probes
> `tcp_keep_alive`: duration - TCP keepalive interval
> `tcp_keep_alive_interval`: duration - TCP keepalive interval between probes
> `udp_fragment`: bool - Enable UDP fragmentation
> `domain_resolver`: object/string - Domain resolver settings
> `network_strategy`: string - Network strategy: "default", "fallback", "hybrid"
> `network_type`: array - Network interface type filter: "wifi", "cellular", "ethernet", "other"
> `fallback_network_type`: array - Fallback network interface types
> `fallback_delay`: duration - Fallback delay when switching network types

## Domain Resolver Options

The `domain_resolver` field can be a simple string (DNS server tag) or an object:

> `server`: string - DNS server tag (required)
> `strategy`: string - DNS resolution strategy: "", "prefer_ipv4", "prefer_ipv6", "ipv4_only", "ipv6_only"
> `disable_cache`: bool - Disable DNS cache for this resolver
> `rewrite_ttl`: number - Rewrite DNS TTL (seconds)
> `client_subnet`: string - EDNS Client Subnet (e.g., "192.168.1.0/24")

### Domain Resolver Examples

Simple form (just server tag):
```json
{
  "domain_resolver": "dns-server-tag"
}
```

Full object form:
```json
{
  "domain_resolver": {
    "server": "dns-server-tag",
    "strategy": "prefer_ipv4",
    "disable_cache": true,
    "rewrite_ttl": 3600,
    "client_subnet": "192.168.1.0/24"
  }
}
```

## Server Options

Outbounds that connect to remote servers have these additional fields:

> `server`: string - Server address (hostname or IP)
> `server_port`: number - Server port (0-65535)

## Network Strategy

The `network_strategy` field controls how multiple network interfaces are used:

- `"default"`: Use the system default network selection
- `"fallback"`: Try primary network type, fall back to others on failure
- `"hybrid"`: Use all available network interfaces simultaneously

## Example

```json
{
  "type": "shadowsocks",
  "tag": "ss-out",
  "server": "example.com",
  "server_port": 8388,
  "method": "2022-blake3-aes-128-gcm",
  "password": "password",
  "connect_timeout": "5s",
  "tcp_fast_open": true,
  "domain_resolver": {
    "server": "dns-local",
    "strategy": "prefer_ipv4"
  }
}
```

# Shared: TLS

TLS configuration options for inbounds and outbounds. Inbound TLS is for server-side configuration, while outbound TLS is for client-side configuration.

## Inbound TLS

Server-side TLS configuration for accepting TLS connections.

> `enabled`: bool - Enable TLS
> `server_name`: string - Server name for SNI
> `insecure`: bool - Skip certificate verification (dangerous, for testing only)
> `alpn`: array - ALPN protocols (e.g., `["h2", "http/1.1"]`)
> `min_version`: string - Minimum TLS version: "1.0", "1.1", "1.2", "1.3"
> `max_version`: string - Maximum TLS version
> `cipher_suites`: array - Cipher suites (Go cipher suite names)
> `curve_preferences`: array - Curve preferences: "P256", "P384", "P521", "X25519", "X25519MLKEM768"
> `certificate`: array - Certificate PEM content (string array)
> `certificate_path`: string - Certificate file path
> `client_authentication`: string - Client auth mode: "no", "request", "require-any", "verify-if-given", "require-and-verify"
> `client_certificate`: array - Trusted client certificates PEM
> `client_certificate_path`: array - Client certificate file paths
> `client_certificate_public_key_sha256`: array - Client certificate public key SHA256 pins
> `key`: array - Private key PEM content
> `key_path`: string - Private key file path
> `kernel_tx`: bool - Enable kernel TLS TX (Linux)
> `kernel_rx`: bool - Enable kernel TLS RX (Linux)
> `acme`: object - ACME (Let's Encrypt) options
> `ech`: object - Encrypted Client Hello options
> `reality`: object - TLS REALITY options

### ACME Options

> `domain`: array - ACME domains to obtain certificates for
> `data_directory`: string - Directory to store ACME data
> `default_server_name`: string - Default SNI for connections without SNI
> `email`: string - Email for ACME account
> `provider`: string - ACME provider (default: Let's Encrypt)
> `disable_http_challenge`: bool - Disable HTTP-01 challenge
> `disable_tls_alpn_challenge`: bool - Disable TLS-ALPN-01 challenge
> `alternative_http_port`: number - Alternative port for HTTP challenge
> `alternative_tls_port`: number - Alternative port for TLS challenge
> `external_account`: object - External account binding
  > `key_id`: string - External account key ID
  > `mac_key`: string - External account MAC key
> `dns01_challenge`: object - DNS-01 challenge configuration
  > `provider`: string - DNS provider: "alidns", "cloudflare", "acmedns"

### Inbound ECH Options

> `enabled`: bool - Enable Encrypted Client Hello
> `key`: array - ECH keys (PEM format)
> `key_path`: string - ECH key file path

### Inbound REALITY Options

> `enabled`: bool - Enable TLS REALITY
> `handshake`: object - Handshake server configuration
  > `server`: string - Handshake server address
  > `server_port`: number - Handshake server port
  > (plus all dialer options)
> `private_key`: string - REALITY private key (base64)
> `short_id`: array - Allowed short IDs
> `max_time_difference`: duration - Maximum allowed time difference

## Outbound TLS

Client-side TLS configuration for connecting to TLS servers.

> `enabled`: bool - Enable TLS
> `disable_sni`: bool - Disable SNI extension
> `server_name`: string - Server name for SNI (overrides connection address)
> `insecure`: bool - Skip certificate verification (dangerous)
> `alpn`: array - ALPN protocols
> `min_version`: string - Minimum TLS version
> `max_version`: string - Maximum TLS version
> `cipher_suites`: array - Cipher suites
> `curve_preferences`: array - Curve preferences
> `certificate`: array - Trusted certificates PEM (CA bundle)
> `certificate_path`: string - Trusted certificates file path
> `certificate_public_key_sha256`: array - Certificate public key SHA256 pins
> `client_certificate`: array - Client certificate PEM
> `client_certificate_path`: array - Client certificate file paths
> `client_key`: array - Client private key PEM
> `client_key_path`: string - Client private key file path
> `fragment`: bool - Enable TLS ClientHello fragmentation
> `fragment_fallback_delay`: duration - Fallback delay for fragmentation
> `record_fragment`: bool - Enable TLS record fragmentation
> `kernel_tx`: bool - Enable kernel TLS TX
> `kernel_rx`: bool - Enable kernel TLS RX
> `ech`: object - Encrypted Client Hello options
> `utls`: object - uTLS fingerprint options
> `reality`: object - TLS REALITY options

### Outbound ECH Options

> `enabled`: bool - Enable Encrypted Client Hello
> `config`: array - ECH configs (base64)
> `config_path`: string - ECH config file path
> `query_server_name`: string - DNS name to query for ECH config

### uTLS Options

uTLS allows mimicking TLS fingerprints of popular browsers:

> `enabled`: bool - Enable uTLS fingerprint
> `fingerprint`: string - TLS fingerprint to use

Available fingerprints:
- `chrome` - Google Chrome (default)
- `firefox` - Mozilla Firefox
- `edge` - Microsoft Edge
- `safari` - Apple Safari
- `ios` - iOS Safari
- `android` - Android
- `random` - Random modern browser fingerprint
- `randomized` - Fully randomized fingerprint

### Outbound REALITY Options

> `enabled`: bool - Enable TLS REALITY
> `public_key`: string - REALITY public key (base64)
> `short_id`: string - REALITY short ID

## Example: Inbound TLS with Certificate

```json
{
  "type": "vless",
  "tag": "vless-in",
```

# Shared: TCP Brutal

TCP Brutal is a congestion control algorithm that provides fixed bandwidth control. It is part of the Multiplex options and requires multiplex to be enabled.

## Overview

TCP Brutal (also known as BBR Brutal) is designed for scenarios where:
- You know your exact bandwidth
- You want consistent, predictable performance
- Traditional congestion control (CUBIC, BBR) is too conservative

## Configuration

TCP Brutal is configured as a sub-object within `multiplex`:

> `brutal`: object - TCP Brutal options
  > `enabled`: bool - Enable TCP Brutal
  > `up_mbps`: number - Upload bandwidth in Mbps
  > `down_mbps`: number - Download bandwidth in Mbps

## Requirements

1. **Multiplex must be enabled**: TCP Brutal requires `multiplex.enabled: true`
2. **Bandwidth must be specified**: Both `up_mbps` and `down_mbps` should match your actual connection speed
3. **Server support**: The server must also support TCP Brutal

## Example: Server (Inbound)

```json
{
  "type": "vless",
  "tag": "vless-in",
  "listen": "::",
  "listen_port": 443,
  "tls": {
    "enabled": true,
    "certificate_path": "/path/to/cert.pem",
    "key_path": "/path/to/key.pem"
  },
  "multiplex": {
    "enabled": true,
    "brutal": {
      "enabled": true,
      "up_mbps": 100,
      "down_mbps": 100
    }
  }
}
```

## Example: Client (Outbound)

```json
{
  "type": "vless",
  "tag": "vless-out",
  "server": "example.com",
  "server_port": 443,
  "tls": {
    "enabled": true
  },
  "multiplex": {
    "enabled": true,
    "brutal": {
      "enabled": true,
      "up_mbps": 50,
      "down_mbps": 100
    }
  }
}
```

## Bandwidth Guidelines

- Set `up_mbps` to your upload speed (client → server)
- Set `down_mbps` to your download speed (server → client)
- For servers, set both to the maximum expected client bandwidth
- Slightly overestimate if unsure to avoid underutilization
- The server's bandwidth limit is applied to each connection, not the total

## Notes

- TCP Brutal is aggressive and may not be suitable for shared networks
- Both client and server must have TCP Brutal enabled for it to work
- If the server doesn't support TCP Brutal, the connection will fall back to standard congestion control

# DNS Configuration

DNS is a core component of sing-box that controls domain resolution behavior. It supports multiple server types, flexible routing rules, and caching.

## DNS Options

> `servers`: array - DNS server list
> `rules`: array - DNS rule list
> `final`: string - Default DNS server tag (required when multiple servers exist)
> `reverse_mapping`: bool - Enable reverse mapping (resolve IP to domain from DNS responses)
> `strategy`: string - DNS strategy: "", "prefer_ipv4", "prefer_ipv6", "ipv4_only", "ipv6_only"
> `disable_cache`: bool - Disable DNS cache
> `disable_expire`: bool - Disable cache expiration
> `independent_cache`: bool - Use independent cache per server
> `cache_capacity`: number - Cache capacity (0 = unlimited)
> `client_subnet`: string - EDNS Client Subnet (e.g., "1.2.3.4/24")

## DNS Server Types

Each server object requires:
> `type`: string - Server type
> `tag`: string - Server tag (unique identifier)

### Local Server
> `type`: "local"

Uses system DNS resolver. No additional fields required.

### UDP Server
> `type`: "udp"
> `server`: string - Server address (domain or IP)
> `server_port`: number - Server port

### TCP Server
> `type`: "tcp"
> `server`: string - Server address
> `server_port`: number - Server port

### TLS Server (DoT)
> `type`: "tls"
> `server`: string - Server address
> `server_port`: number - Server port
> `tls`: object - TLS options (see TLS configuration)

### HTTPS Server (DoH)
> `type`: "https"
> `server`: string - Server address
> `server_port`: number - Server port
> `tls`: object - TLS options
> `path`: string - HTTP path (default: "/dns-query")
> `method`: string - HTTP method ("GET" or "POST")
> `headers`: object - HTTP headers

### HTTP/3 Server (DoH3)
> `type`: "h3"
> `server`: string - Server address
> `server_port`: number - Server port
> `tls`: object - TLS options
> `path`: string - HTTP path

### DHCP Server
> `type`: "dhcp"
> `interface`: string - Network interface to use

### FakeIP Server
> `type`: "fakeip"
> `inet4_range`: string - IPv4 range (default: "198.18.0.0/15")
> `inet6_range`: string - IPv6 range (default: "fc00::/18")

FakeIP returns fake IP addresses for DNS queries, useful for gaming and avoiding DNS leaks.

### Hosts Server
> `type`: "hosts"
> `path`: array - Hosts file paths
> `predefined`: object - Predefined domain-to-address mappings

### Tailscale Server
> `type`: "tailscale"
> `endpoint`: string - Tailscale endpoint tag
> `accept_default_resolvers`: bool - Accept default resolvers from Tailscale

### Common Dialer Options (for remote servers)
> `detour`: string - Outbound tag for connection
> `bind_interface`: string - Bind to interface
> `inet4_bind_address`: string - IPv4 bind address
> `inet6_bind_address`: string - IPv6 bind address

## DNS Rules

DNS rules match conditions and execute actions. Rules are evaluated in order; first match wins.

### Rule Conditions

> `inbound`: array - Match inbound tags
> `ip_version`: number - Match IP version (4 or 6)
> `query_type`: array - DNS query types ("A", "AAAA", "MX", "TXT", etc.)
> `network`: array - Network types ("tcp", "udp")
> `auth_user`: array - Authenticated users
> `protocol`: array - Protocols
> `domain`: array - Exact domain matches
> `domain_suffix`: array - Domain suffixes (e.g., [".google.com"])
> `domain_keyword`: array - Domain keywords
> `domain_regex`: array - Domain regex patterns
> `geosite`: array - Geosite categories
> `source_geoip`: array - Source GeoIP codes
> `geoip`: array - GeoIP codes
> `ip_cidr`: array - IP CIDR ranges
> `ip_is_private`: bool - Match private IPs
> `ip_accept_any`: bool - Accept any IP
> `source_ip_cidr`: array - Source IP CIDR
> `source_ip_is_private`: bool - Source is private IP
> `source_port`: array - Source ports
> `source_port_range`: array - Source port ranges
> `port`: array - Destination ports
> `port_range`: array - Destination port ranges
> `process_name`: array - Process names
> `process_path`: array - Process paths
> `process_path_regex`: array - Process path regex
> `package_name`: array - Android package names
> `user`: array - User names
> `user_id`: array - User IDs
> `outbound`: array - Outbound tags
> `clash_mode`: string - Clash mode ("rule", "global", "direct")
> `network_type`: array - Network types
> `network_is_expensive`: bool - Expensive network (metered)
> `network_is_constrained`: bool - Constrained network
> `wifi_ssid`: array - WiFi SSIDs
> `wifi_bssid`: array - WiFi BSSIDs
> `rule_set`: array - Rule set tags
> `rule_set_ip_cidr_match_source`: bool - Match source IP in rule set
> `rule_set_ip_cidr_accept_empty`: bool - Accept empty IP
> `invert`: bool - Invert match result

### Logical Rules

> `type`: "logical"
> `mode`: "and" or "or"
> `rules`: array - Nested rules

### DNS Rule Actions

#### Route Action
> `action`: "route"
> `server`: string - DNS server tag to use

Routes the query to specified DNS server.

#### Route-Options Action
> `action`: "route-options"
> `strategy`: string - Override DNS strategy
> `disable_cache`: bool - Disable cache for this query
> `rewrite_ttl`: number - Rewrite TTL value
> `client_subnet`: string - Override EDNS Client Subnet

> `action`: "reject"
> `method`: string - Reject method: "default", "drop", "reply"

- `default`: Returns REFUSED response
- `drop`: Silently drops the query
- `reply`: Returns empty response

#### Predefined Action
> `action`: "predefined"
> `rcode`: string - DNS response code
> `answer`: array - Answer records
> `ns`: array - NS records
> `extra`: array - Extra records

Returns predefined DNS response.

# Rule Set

Rule sets allow organizing and reusing routing rules. They can be defined inline, loaded from local files, or downloaded from remote URLs.

## Rule Set Types

> `type`: string - "inline", "local", or "remote"
> `tag`: string - Required unique tag
> `format`: string - "source" or "binary" (auto-detected from extension: .json=source, .srs=binary)

## Inline Rule Set

> `type`: "inline"
> `rules`: array - Headless rules

Inline rule sets are defined directly in configuration.

```json
{
  "tag": "my-inline-rules",
  "type": "inline",
  "rules": [
    {

    {
      "ip_cidr": ["192.168.100.0/24"]
    }
  ]
}
```

## Local Rule Set

> `type`: "local"
> `path`: string - File path to rule set

```json

  "tag": "local-rules",

}
```

## Remote Rule Set

> `type`: "remote"
> `url`: string - Download URL

```json
  "url": "https://example.com/rule-sets/geosite-google.srs",

  "update_interval": "12h"
}
```

## Headless Rule Format

Headless rules define only conditions (no actions). They are used in rule sets.

### Available Condition Fields

> `query_type`: array - DNS query types
> `network`: array - Network types ("tcp", "udp")
> `domain`: array - Exact domains
> `domain_suffix`: array - Domain suffixes
> `domain_keyword`: array - Domain keywords
> `domain_regex`: array - Domain regex patterns
> `source_ip_cidr`: array - Source IP CIDR
> `ip_cidr`: array - Destination IP CIDR
> `source_port`: array - Source ports
> `source_port_range`: array - Source port ranges
> `port`: array - Destination ports
> `port_range`: array - Destination port ranges
> `process_name`: array - Process names
> `process_path`: array - Process paths
> `process_path_regex`: array - Process path regex
> `package_name`: array - Android package names
> `network_type`: array - Network types
> `network_is_expensive`: bool - Metered network
> `network_is_constrained`: bool - Constrained network
> `wifi_ssid`: array - WiFi SSIDs
> `wifi_bssid`: array - WiFi BSSIDs
> `network_interface_address`: object - Interface address
> `default_interface_address`: array - Default interface addresses
> `invert`: bool - Invert match

## Rule Set Source Format

The source format is a JSON file with version 4:

```json
{
  "version": 4,
  "rules": [
    {
      "domain_suffix": [".google.com", ".googleapis.com"]
    },
    {
      "domain_keyword": ["youtube"]
    },
    {
      "ip_cidr": ["142.250.0.0/15"]
    }
  ]
}
```

**Note**: In source format, the `outbound` field in rules is ignored. Rule sets only define match conditions.

## Binary Format

The binary format (.srs) is compiled from source format for:
- Smaller file size
- Faster loading
- Lower memory usage

Use `sing-box rule-set compile` to convert source to binary.

## Using Rule Sets in Route Rules

```json
{

    "rules": [

      }

      {
        "tag": "google",

  }
```

# Inbound: TUN

TUN is the most important inbound for transparent proxy on mobile and desktop platforms. It creates a virtual network interface to intercept all traffic.

## Basic Configuration

> `type`: "tun"
> `tag`: string - Inbound tag

## Interface Settings

> `interface_name`: string - TUN interface name (default: "sing-box")
> `mtu`: number - MTU size (default: 9000)
> `address`: array - IP addresses with CIDR (e.g., ["172.19.0.1/30", "fdfe:dcba:9876::1/126"])

## Routing Settings

> `auto_route`: bool - Auto configure system routes
> `iproute2_table_index`: number - iproute2 routing table index (Linux)
> `iproute2_rule_index`: number - iproute2 rule index (Linux)
> `auto_redirect`: bool - Auto redirect mode (Linux only)
> `auto_redirect_input_mark`: number - Input mark for auto redirect
> `auto_redirect_output_mark`: number - Output mark for auto redirect
> `auto_redirect_reset_mark`: number - Reset mark for auto redirect
> `auto_redirect_nfqueue`: number - NFQUEUE number for auto redirect
> `auto_redirect_iproute2_fallback_rule_index`: number - Fallback rule index
> `exclude_mptcp`: bool - Exclude MPTCP connections
> `loopback_address`: array - Custom loopback addresses

## Strict Route

> `strict_route`: bool - Strict route mode

When enabled, blocks connections that cannot be handled by sing-box (prevents leaks).

## Route Address Control

> `route_address`: array - Extra route addresses
> `route_address_set`: array - Route address rule sets
> `route_exclude_address`: array - Addresses to exclude from routing
> `route_exclude_address_set`: array - Exclude address rule sets

> `exclude_interface`: array - Exclude these interfaces
> `include_uid`: array - Include only these UIDs
> `include_uid_range`: array - UID ranges to include
> `exclude_uid`: array - Exclude these UIDs

> `include_android_user`: array - Android user IDs to include
> `include_package`: array - Android packages to include

> `udp_timeout`: duration - UDP connection timeout

> `platform`: object

### Windows

> `windows.endpoint_independent_nat`: bool

# Inbound: Shadowsocks

Shadowsocks inbound accepts Shadowsocks protocol connections, supporting both legacy and 2022 encryption methods.

## Shadowsocks Inbound

> `type`: "shadowsocks"
> `tag`: string - Unique identifier for the inbound
> `listen`: string - Listen address
> `listen_port`: number - Listen port
> `tcp_fast_open`: bool - Enable TCP Fast Open
> `udp_fragment`: bool - Enable UDP fragmentation
> `network`: string/array - Network protocols: "tcp", "udp", or ["tcp", "udp"] (default: ["tcp", "udp"])
> `method`: string - Encryption method
> `password`: string - Pre-shared key/password
> `users`: array - Multi-user support (for 2022 methods)
  > `name`: string - Username
  > `password`: string - User password/key
> `destinations`: array - Shadowsocks relay destinations
  > `name`: string - Destination name
  > `password`: string - Destination password
  > `server`: string - Server address
  > `server_port`: number - Server port
> `multiplex`: object - Inbound multiplex configuration
> `managed`: bool - Enable managed mode for multi-user configurations

## Encryption Methods

### 2022 Methods (Recommended)
- `2022-blake3-aes-128-gcm`
- `2022-blake3-aes-256-gcm`
- `2022-blake3-chacha20-poly1305`

These methods use 16-byte or 32-byte keys (base64-encoded).

### Legacy Methods
- `aes-128-gcm`
- `aes-256-gcm`
- `chacha20-ietf-poly1305`
- `xchacha20-ietf-poly1305`
- `none` (no encryption, for testing only)

> `type`: "direct"
> `tag`: string - Unique identifier for the inbound
> `listen`: string - Listen address
> `listen_port`: number - Listen port
> `network`: string/array - Network protocols: "tcp", "udp", or ["tcp", "udp"]
> `override_address`: string - Override destination address
> `override_port`: number - Override destination port

### Direct Inbound Example

```json
{
  "type": "direct",
  "tag": "direct-in",
  "listen": "127.0.0.1",
  "listen_port": 8080,
  "network": "tcp"
}
```

## ShadowTLS Inbound

ShadowTLS wraps another proxy protocol with TLS handshake masquerading.

> `type`: "shadowtls"
> `tag`: string - Unique identifier for the inbound
> `listen`: string - Listen address
> `listen_port`: number - Listen port
> `version`: number - ShadowTLS version (1, 2, or 3)
> `password`: string - Server password
> `users`: array - Multi-user support (version 3 only)
  > `name`: string - Username
  > `password`: string - User password
> `handshake`: object - TLS handshake server configuration
  > `server`: string - Handshake server address
  > `server_port`: number - Handshake server port
> `handshake_for_server_name`: object - Per-SNI handshake configuration
> `strict_mode`: bool - Enable strict mode
> `wildcard_sni`: string - Wildcard SNI handling: "off", "authed", "all"

### ShadowTLS Inbound Example

```json
{
  "type": "shadowtls",
  "tag": "shadowtls-in",
  "listen": "0.0.0.0",
  "listen_port": 443,
  "version": 3,
  "password": "server-password",
  "handshake": {
    "server": "www.microsoft.com",
    "server_port": 443
  }
}
```

# Outbound: Direct / Block / SOCKS / HTTP

## Direct Outbound

The direct outbound sends traffic directly to the destination without any proxy. Useful for bypassing certain traffic or as a fallback.

> `type`: "direct"
> `tag`: string - Unique identifier for this outbound

Plus [dialer options](/shared-dialer) only.

**Note:** `override_address` and `override_port` fields have been removed. Use `route` action with `override_address`/`override_port` instead.

### Example

```json
{
  "type": "direct",
  "tag": "direct-out"
}
```

## Block Outbound

The block outbound drops all traffic. Connections will be rejected.

> `type`: "block"
> `tag`: string - Unique identifier for this outbound

No additional options.

**Deprecated:** Use route rule action `"reject"` instead. The block outbound may be removed in future versions.

### Example

```json
{
  "type": "block",
  "tag": "block-out"
}
```

# Outbound: Shadowsocks

Shadowsocks protocol outbound supporting multiple encryption methods and plugins.

> `type`: "shadowsocks"
> `tag`: string - Unique identifier for this outbound

Plus [dialer options](/shared-dialer), plus [server options](/shared-server) (server, server_port), plus:

> `method`: string - Encryption method (e.g., "2022-blake3-aes-128-gcm", "aes-128-gcm", "chacha20-ietf-poly1305", "none")
> `password`: string - Password or base64-encoded key
> `plugin`: string - SIP003 plugin name (e.g., "obfs-local", "v2ray-plugin")
> `plugin_opts`: string - Plugin options string
> `network`: string/array - Network protocols: "tcp", "udp", or ["tcp", "udp"]
> `udp_over_tcp`: object/bool - UDP over TCP configuration for servers that don't support UDP
> `multiplex`: object - [Outbound multiplex options](/shared-multiplex)
> `tls`: object - [Outbound TLS](/shared-tls) configuration (for ShadowTLS-wrapped servers)

### Encryption Methods

**2022 Methods (Recommended):**
- `2022-blake3-aes-128-gcm`
- `2022-blake3-aes-256-gcm`
- `2022-blake3-chacha20-poly1305`

**Legacy AEAD Methods:**
- `aes-128-gcm`
- `aes-256-gcm`
- `chacha20-ietf-poly1305`

**Legacy Stream Methods (Not Recommended):**
- `aes-128-ctr`
- `aes-192-ctr`
- `aes-256-ctr`
- `aes-128-cfb`
- `rc4-md5`
- `none`

### Example

```json
{
  "type": "shadowsocks",
  "tag": "ss-out",
  "server": "ss.example.com",
  "server_port": 8388,
  "method": "2022-blake3-aes-128-gcm",
  "password": "base64-encoded-key-here"
}
```

### Example with Plugin

```json
{
  "type": "shadowsocks",
  "tag": "ss-plugin-out",
  "server": "ss.example.com",
  "server_port": 8388,
  "method": "aes-128-gcm",
  "password": "password",
  "plugin": "obfs-local",
  "plugin_opts": "obfs=http;obfs-host=www.bing.com"
}
```

### Example with ShadowTLS

```json
{
  "type": "shadowsocks",
  "tag": "ss-shadowtls-out",
  "server": "shadowtls.example.com",
  "server_port": 443,
  "method": "aes-128-gcm",
  "password": "password",
  "tls": {
    "enabled": true,
    "server_name": "www.example.com"
  }
}
```

# Outbound: VMess / VLESS

> `authenticated_length`: bool - Enable authenticated length (default: false)
> `network`: string/array - Network protocols: "tcp", "udp"
> `tls`: object - [Outbound TLS](/shared-tls) configuration
> `packet_encoding`: string - Packet encoding: "none", "packetaddr"
> `multiplex`: object - [Multiplex options](/shared-multiplex)
> `transport`: object - [Transport options](/shared-transport)

### Example

```json
{
  "type": "vmess",
```

# Outbound: Hysteria / Hysteria2

Hysteria protocols are high-performance proxy protocols based on QUIC, designed for high-latency and high-loss networks.

Hysteria 1 protocol (original version).

> `type`: "hysteria"

> `server_ports`: array - Port hopping list (e.g., ["2000-5000"])
> `hop_interval`: duration - Port hop interval (default: "30s")
> `up`: string/number - Upload bandwidth (e.g., "100 Mbps", 100000000)
> `up_mbps`: number - Upload bandwidth in Mbps
> `down`: string/number - Download bandwidth (e.g., "100 Mbps", 100000000)
> `down_mbps`: number - Download bandwidth in Mbps
> `obfs`: string - Obfuscation password
> `auth`: string - Authentication string (base64 encoded)
> `auth_str`: string - Authentication string (plain text)
> `recv_window_conn`: number - Receive window per connection
> `recv_window`: number - Global receive window
> `disable_mtu_discovery`: bool - Disable path MTU discovery
> `network`: string/array - Network protocols: "tcp", "udp"
> `tls`: object - [Outbound TLS](/shared-tls) configuration (**required**)

### Example

```json
{
  "type": "hysteria",
  "tag": "hysteria-out",
  "server": "hysteria.example.com",
  "server_port": 443,
  "up_mbps": 100,
  "down_mbps": 100,
  "auth_str": "password",
  "tls": {
    "enabled": true,
    "server_name": "hysteria.example.com",
    "insecure": true
  }
}
```

> `tag`: string - Unique identifier for this outbound

Plus [dialer options](/shared-dialer), plus:

> `executable_path`: string - Path to Tor executable (default: "tor")
> `extra_args`: array - Extra command-line arguments for Tor
> `data_directory`: string - Tor data directory path
> `torrc`: object - Tor configuration as key-value pairs

### Example

```json
{
  "type": "tor",
  "tag": "tor-out",
  "executable_path": "/usr/bin/tor",
  "data_directory": "/var/lib/tor",
  "torrc": {
    "ControlPort": "9051"
  }
}
```

## SSH Outbound

SSH protocol outbound for port forwarding over SSH.

> `type`: "ssh"
> `tag`: string - Unique identifier for this outbound

Plus [dialer options](/shared-dialer), plus [server options](/shared-server), plus:

> `user`: string - SSH username
> `password`: string - SSH password
> `private_key`: array - Private key PEM content (lines)
> `private_key_path`: string - Path to private key file
> `private_key_passphrase`: string - Passphrase for encrypted private key
> `host_key`: array - Known host keys (for verification)
> `host_key_algorithms`: array - Allowed host key algorithms
> `client_version`: string - SSH client version string

### Example

```json
{
  "type": "ssh",
  "tag": "ssh-out",
  "server": "ssh.example.com",
  "server_port": 22,
  "user": "user",
  "private_key_path": "/home/user/.ssh/id_rsa"
}
```

## Selector Outbound

Selector allows manual switching between multiple outbounds.

> `type`: "selector"
> `tag`: string - Unique identifier for this outbound
> `outbounds`: array - List of outbound tags to select from
> `default`: string - Default outbound tag
> `interrupt_exist_connections`: bool - Interrupt existing connections on switch (default: false)

### Example

```json
{
```

> `type`: "urltest"

# Endpoint: WireGuard / Tailscale

Endpoints are similar to outbounds but represent network endpoints like VPN tunnels. They create virtual network interfaces and handle all traffic through them.

## WireGuard Endpoint

WireGuard VPN endpoint. Since sing-box 1.11.0, WireGuard is an endpoint, not an outbound.

> `type`: "wireguard"
> `tag`: string - Unique identifier for this endpoint

Plus [dialer options](/shared-dialer), plus:

> `system`: bool - Use system WireGuard (requires root/CAP_NET_ADMIN)
> `name`: string - Interface name (e.g., "wg0")
> `mtu`: number - MTU size (default: 1280)
> `address`: array - IP addresses with CIDR (e.g., ["10.0.0.2/24", "fd00::2/64"])
> `private_key`: string - WireGuard private key (base64)
> `listen_port`: number - Local UDP port to listen on
> `peers`: array - List of WireGuard peers
> `udp_timeout`: duration - UDP timeout (default: "5m")
> `workers`: number - Number of workers (default: CPU count)

### Peer Object

> `address`: string - Peer endpoint IP or hostname
> `port`: number - Peer endpoint port
> `public_key`: string - Peer public key (base64)
> `pre_shared_key`: string - Pre-shared key (base64, optional)
> `allowed_ips`: array - CIDR ranges to route through this peer
> `persistent_keepalive_interval`: number - Keepalive interval in seconds
> `reserved`: array - Reserved bytes (3 numbers: [client, server, 0])

### Example

```json
{
  "type": "wireguard",
  "tag": "wg-endpoint",
  "address": ["10.0.0.2/24"],
  "private_key": "aBcDeFgHiJkLmNoPqRsTuVwXyZ1234567890ABCDEF=",
  "peers": [
    {
      "address": "wg.example.com",
      "port": 51820,
      "public_key": "XyZ1234567890ABCDEFaBcDeFgHiJkLmNoPqRsTuVw=",
      "pre_shared_key": "1234567890ABCDEFaBcDeFgHiJkLmNoPqRsTuVwXyZ=",
      "allowed_ips": ["0.0.0.0/0", "::/0"],
      "persistent_keepalive_interval": 25
    }
  ],
  "mtu": 1280
}
```

## Tailscale Endpoint

Tailscale VPN endpoint for connecting to Tailscale networks.

> `type`: "tailscale"
> `tag`: string - Unique identifier for this endpoint

Plus [dialer options](/shared-dialer), plus:

> `state_directory`: string - Directory for Tailscale state files
> `auth_key`: string - Tailscale auth key
> `control_url`: string - Custom control server URL
> `ephemeral`: bool - Ephemeral node (disappears on disconnect)
> `hostname`: string - Node hostname
> `accept_routes`: bool - Accept routes advertised by other nodes
> `exit_node`: string - Exit node IP or hostname to use
> `exit_node_allow_lan_access`: bool - Allow LAN access when using exit node
> `advertise_routes`: array - CIDR ranges to advertise
> `advertise_exit_node`: bool - Advertise as exit node
> `advertise_tags`: array - Tags to advertise (e.g., ["tag:server"])
> `relay_server_port`: number - DERP relay server port
> `relay_server_static_endpoints`: array - Static relay endpoints (IP:port strings)
> `system_interface`: bool - Use system Tailscale interface
> `system_interface_name`: string - System interface name
> `system_interface_mtu`: number - System interface MTU
> `udp_timeout`: duration - UDP timeout

### Example

```json
{
  "type": "tailscale",
  "tag": "tailscale-ep",
  "auth_key": "tskey-auth-xxxxxxxx",
  "hostname": "sing-box-node",
  "accept_routes": true
}
```

> `tls`: object - [Inbound TLS](/shared-tls) configuration

> `mesh_psk`: string - Mesh pre-shared key
> `mesh_psk_file`: string - Path to mesh pre-shared key file
> `stun`: bool/number/object - STUN listener configuration

### Example

# Gotchas & Common Mistakes

Common pitfalls and important notes when configuring sing-box.

## 1. Unknown Fields Are Rejected

sing-box uses `DisallowUnknownFields()` for JSON parsing. Typos in field names will cause **hard errors**, not be silently ignored. Always double-check field names against documentation.

```json
// Wrong - will error
{
  "typ": "socks"  // should be "type"
}

// Wrong - will error
{
  "type": "socks",
  "sever": "1.2.3.4"  // should be "server"
}
```

## 2. Tags Must Be Unique Across All Inbounds AND Outbounds

Tags share a single namespace across inbounds, outbounds, and endpoints. Using the same tag twice will cause errors.

```json
// Wrong - duplicate tag
{
  "inbounds": [
    { "type": "socks", "tag": "proxy" }
  ],
  "outbounds": [
    { "type": "vless", "tag": "proxy" }  // Error: tag already used
  ]
}
```

## 3. Block Outbound Deprecated

The block outbound is deprecated. Use route rule action instead:

```json
// Old (deprecated)
{
  "outbounds": [
    { "type": "block", "tag": "block" }
  ],
  "route": {
    "rules": [
      { "outbound": "block" }
    ]
  }
}

// New (recommended)
{
  "route": {
    "rules": [
      { "action": "reject" }
    ]
  }
}
```

## 4. DNS Outbound Deprecated

The DNS outbound is deprecated. Use route rule action instead:

```json
// Old (deprecated)
{
  "outbounds": [
    { "type": "dns", "tag": "dns-out" }
  ],
  "route": {
    "rules": [
      { "protocol": "dns", "outbound": "dns-out" }
    ]
  }
}

// New (recommended)
{
  "route": {
    "rules": [
      { "protocol": "dns", "action": "hijack-dns" }
    ]
  }
}
```

## 5. Direct Outbound Override Fields Removed

`override_address` and `override_port` on direct outbound have been removed. Use route action instead:

```json
// Old (removed)
{
  "outbounds": [
    {
      "type": "direct",
      "tag": "dns-override",
      "override_address": "8.8.8.8",
      "override_port": 53
    }
  ]
}

// New (use route action)
{
  "route": {
    "rules": [
      {
        "protocol": "dns",
        "action": "resolve",
        "override_address": "8.8.8.8",
        "override_port": 53
      }
    ]
  }
}
```

## 6. WireGuard Is Now an Endpoint

Since sing-box 1.11.0, WireGuard is an **endpoint**, not an outbound. Update your configurations:

```json
// Wrong (old format)
{
  "outbounds": [
    {
      "type": "wireguard",
      "tag": "wg",
      ...
    }
  ]
}

// Correct (new format)
{
  "endpoints": [
    {
      "type": "wireguard",
      "tag": "wg",
      ...
    }
  ]
}
```

## 7. Legacy Inbound Fields Removed

Fields like `sniff`, `domain_strategy`, `udp_timeout` etc. have been removed from inbounds. These must now be configured in route rules or route options:

```json
// Wrong (old format)
{
  "inbounds": [
    {
      "type": "tun",
      "sniff": true,
      "domain_strategy": "prefer_ipv4"
    }
  ]
}

// Correct (new format)
{
  "inbounds": [
    {
      "type": "tun",
      ...
    }
  ],
  "route": {
    "rules": [
      {
        "action": "sniff"
      },
      {
        "action": "resolve",
        "strategy": "prefer_ipv4"
      }
    ]
  }
}
```

## 8. TLS Required for Many Protocols

These protocols **require** TLS configuration:
- VLESS
- Trojan
- Hysteria2
- TUIC
- AnyTLS
- Naive

Without TLS, these outbounds will fail.

## 9. FwMark Accepts Hex Strings

The `fwmark` field accepts hex strings with `0x` prefix:

```json
{
  "route": {
    "rules": [
      {
        "action": "route-options",
        "fwmark": "0xff"  // hex string
      }
    ]
  }
}
```

## 10. UDP Timeout Accepts Seconds as Number

For backward compatibility, integer values for `udp_timeout` are treated as seconds:

```json
{
  // Both work:
  "udp_timeout": 300      // 300 seconds (legacy format)
  "udp_timeout": "5m"     // 5 minutes (recommended format)
}
```

## 11. Domain Resolver Can Be a String

The `domain_resolver` field can be just a server tag string, or a full object:

```json
{
  // Simple form - just the tag
  "domain_resolver": "google-dns"

  // Full form
  "domain_resolver": {
    "server": "google-dns",
    "rewrite_ttl": 300
  }
}
```

## 12. Rule Set Format Auto-Detection

Rule set format is auto-detected by file extension:
- `.json` = source format (JSON)
- `.srs` = binary format (compiled)

```json
{
  "rule_set": [
    {
      "tag": "geoip-cn",
      "type": "remote",
      "format": "binary",  // Explicit, but auto-detected from .srs
      "url": "https://example.com/geoip-cn.srs"
    }
  ]
}
```

## 13. TUN Requires Special Permissions

Running TUN inbound requires elevated permissions:
- **Linux**: root or `CAP_NET_ADMIN` capability
- **macOS**: root
- **Windows**: Administrator or RunAsService
- **Android/iOS**: VPN permission (system prompt)

## 14. Config Merging

When using multiple config files, they are deep-merged:
- Objects are merged recursively
- Arrays are concatenated (not replaced)

```bash
# Merge multiple configs
sing-box run -c base.json -c overrides.json
```

## 15. SIGHUP Reload

Sending `SIGHUP` to a running sing-box process triggers config reload:

```bash
# Reload configuration
kill -HUP <pid>

# Or using pkill
pkill -HUP sing-box
```

## 16. TLS Fragment and TLS Record Fragment Are Mutually Exclusive

You cannot use both `tls_fragment` and `tls_record_fragment` in the same `route-options` action:

```json
// Wrong - mutually exclusive
{
  "route": {
    "rules": [
      {
        "action": "route-options",
        "tls_fragment": true,
        "tls_record_fragment": true  // Error!
      }
    ]
  }
}

// Correct - choose one
{
  "route": {
    "rules": [
      {
        "action": "route-options",
        "tls_fragment": true
      }
    ]
  }
}
```
